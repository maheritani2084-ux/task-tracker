## Feature 1: Due date+

Using GitHub Copilot Chat as my AI agent tool helped me move faster during the overdue-date feature work, especially when I needed quick implementation support and test scaffolding. It was especially useful for turning a broad requirement into concrete code changes in the FastAPI backend, the frontend modal, and the pytest suite. I used it to generate test cases, propose frontend refactors, and quickly check whether my assumptions about status filters, due-date validation, and overdue logic were consistent with the existing architecture. In several moments, it accelerated the workflow by surfacing useful patterns and reducing the amount of boilerplate I had to write by hand.

There were also moments when it slowed me down. The biggest issue was that the tool sometimes suggested changes too confidently without fully respecting the project’s existing constraints. For example, it initially proposed frontend logic that could have introduced unnecessary behavior changes, or it would suggest implementation details that did not align with the app’s current structure. That required extra review time to confirm what should be preserved, such as the backend-driven overdue computation, the existing pill styling, and the current request/response shapes. In other words, the AI was helpful for speed, but not always reliable enough to follow blindly.

One place where my review changed the result was the overdue pill styling and rendering. The AI suggested a more general “make it look better” approach, but that would have been too vague and could have introduced regressions. I reviewed the request against the existing pill rules, the card layout, and the accessibility requirements, and then refined it into a narrower change that preserved the same geometry, visible label, aria-label, and behavior. That review step made the final result more accurate, safer, and easier to validate. Overall, the tool was valuable as a co-pilot, but the final quality depended on careful human review and constraint checking.



## Feature 2: tags/labels

For Feature 2, I used GitHub Copilot Chat as my AI agent tool to help turn a broad requirement into concrete implementation steps. It was especially useful when I needed to move from the idea of tag support to the actual backend and frontend work. The tool helped me quickly sketch the structure for tag validation, the task model changes, the modal updates, and the test cases needed for create, update, and filter behavior. That saved time because it reduced the amount of repetitive scaffolding I had to write by hand and gave me a strong starting point to refine.

There were also moments when it slowed me down. The biggest issue was that the tool sometimes answered too confidently without fully respecting the project’s existing constraints. For example, when I asked for help with the tag filter behavior, the first pass did not fully reflect the requirement that partial matching should work for values like “work” even when the stored tags were expressed as a compound string such as “work, urgent.” That meant I had to stop, reframe the request, and tighten the acceptance criteria before moving forward. In that sense, the AI was helpful for speed, but it still required careful steering.

One place where my review changed the result was the tag filter implementation. The initial suggestion was too general and could have led to an overbuilt fix. I reviewed it against the existing frontend flow, the backend API contract, and the expected behavior for tag preservation and partial matches. I then narrowed the change so it only affected the filter logic and preserved the rest of the board behavior. That review step made the final result more accurate, safer, and easier to verify.
